# 实时说话人识别与详情工作台修复设计

日期：2026-07-14

## 1. 背景与目标

实时会议详情当前有五个用户可见问题：

1. ASR 会把会议标题（例如“快速会议 7-14 16:00”）错误写入逐字稿。
2. 多人讲话时所有片段都显示“实时发言人”，无法稳定区分发言人；已登记声纹也没有显示真实姓名。
3. 详情头部显示 `Mode / Transcript / Recognition / Sensitive / Template / Voiceprint` 调试条，对业务用户无意义。
4. 逐字稿搜索、发言人筛选和保存状态使用英文，且占用空间过大。
5. 右侧五个 AI 工具按钮自动换行，视觉层级和点击区域不稳定。

本次只修改实时会议的转写身份链路以及实时/导入共用详情工作台的可见控件，不改变导入转写台账职责，也不把实时和导入数据重新耦合。

## 2. 已确认的交互规格

### 2.1 逐字稿工具栏

用户选择保留图三能力：

- 搜索框文案改为“搜索转写内容”。
- 发言人筛选默认项改为“全部发言人”。
- 保存状态改为“已保存 / 保存中 / 保存失败”。
- 三个控件保持单行紧凑排列；窄屏允许合理换行，但不能遮挡编辑工具栏。

### 2.2 调试信息条

删除详情页中的会议配置调试条及其渲染调用。处理模式、转写版本和模型状态仍保留在后端数据与日志中，不再作为普通用户界面展示。

### 2.3 AI 工具导航

- 五个按钮固定为单行等宽布局：`规整 / 摘要 / 纪要 / 待办 / 标记`。
- 收起按钮绝对定位在右栏边缘，不参与五列网格计算。
- 桌面端五列均分可用宽度；窄屏保持最小点击宽度并横向滚动，不允许第五个按钮单独掉到下一行。
- 活跃态、运行态 spinner 和禁用态继续可见，且不能改变按钮尺寸。

## 3. 会议标题污染修复

### 3.1 根因

`build_realtime_context()` 当前把 `meetingName/title/fileName` 与参与人和识别词库一起写入 DashScope `corpus.text`。上下文偏置本应帮助专有名词识别，但会议标题是一个完整、强提示短语；静音、短语音或低信息量音频下，模型会直接复述它。

### 3.2 修复规则

1. 实时上下文不再包含会议标题和导入文件名，只保留参与人姓名及冻结的识别优化词。
2. 导入 ASR 的既有词库策略不因本次修改改变；实时上下文使用显式参数或独立构造函数表达“不含标题”。
3. 在实时最终片段持久化前增加防御性检查：
   - 去除空白和常见末尾标点后，若文本与当前会议标题完全相同，则判定为上下文回声。
   - 上下文回声返回结构化 `status`，不写入 segments，不增加 transcript revision。
   - 只过滤“整段仅为标题”的结果；正常句子中确实提到会议标题时不做字符串删除，避免破坏真实发言。
4. `rawText` 仍保留在诊断事件中，但不能进入用户逐字稿或后续 AI 工具输入。

## 4. 实时说话人和声纹识别

### 4.1 根因

当前 DashScope 原生流式路径在 `handle_stream_event()` 收到最终文本后直接调用 `_finalize_realtime_transcript_event()`。该路径没有把对应 PCM 音频交给本地 diarization/声纹服务，因此 `realtime_stream.py` 只能写死 `speakerName="实时发言人"`。旧的同步 WAV 分片路径虽调用了 diarization，但并不是当前正常主路径。

### 4.2 方案选择

采用“流式 ASR + 每个最终语音段的声纹 embedding 跟踪”：

- 不采用 UI 交替编号，因为它无法反映真实说话人。
- 不在每个片段后重新对整场录音做 diarization，因为处理量会随会议时长持续增长，进一步拖慢实时结果。
- 对每个最终语音段提取一次 CAM++ embedding，在当前会议内做增量聚类；这既能稳定区分未知发言人，也能优先匹配声纹库姓名。

### 4.3 音频与时间轴

1. 浏览器继续向后端发送单声道 PCM16 帧，DashScope partial/final 速度不受影响。
2. 后端 WebSocket 会话保存有界 PCM 环形缓冲，并记录缓冲起点对应的音频毫秒。
3. DashScope 最终事件提供或推算 `startMs/endMs` 后，从缓冲中截取该区间并封装为临时 16k WAV。
4. 环形缓冲只保留完成识别和说话人分析所需的最近音频，不能无限累积整场会议内存。
5. 临时 WAV 在分析结束后删除；删除仅针对本次创建的单个明确文件路径，遵守仓库禁止批量/递归删除的约束。

### 4.4 身份决策顺序

每个最终片段按以下顺序决定显示身份：

1. **声纹库命中**：CAM++ 与已登记人员相似度达到现有生产阈值，显示声纹库姓名，并带部门/职位与 `voiceprintId`。
2. **会议内已知聚类命中**：未命中声纹库时，将 embedding 与当前会议未知说话人质心比较；超过聚类阈值则复用对应编号。
3. **新说话人**：没有已有聚类达到阈值时创建新质心，按首次出现顺序分配“发言人1、发言人2……” 。
4. embedding/模型服务暂时不可用时，保留文本并使用稳定的降级编号；不能回退成所有片段相同的“实时发言人”。

声纹姓名优先于匿名编号。同一个匿名聚类后续若命中声纹库，应更新该聚类身份，并同步修正当前会议中该聚类的历史片段显示名。

### 4.5 模型服务接口

为避免把敏感 embedding 暴露给浏览器，新增仅供业务后端调用的本地模型接口：

```text
POST /v1/speakers/embedding
body: { audio_path }
response: { model, embedding, realModel, fallbackReason }
```

业务后端 `LocalVoiceprintClient` 增加对应方法。embedding 只存在于后端会话内存，不写入前端响应；需要持久化的只有聚类 ID、显示姓名、来源和置信度。

### 4.6 前端更新

- 最终 transcript 事件立即追加文本；说话人分析完成后可通过 `speaker_update` 事件更新指定 segment，避免阻塞 partial 文本显示。
- `speaker_update` 必须包含 `meetingId/sessionToken/segmentId`，前端只更新当前实时会议匹配的片段。
- 发言人列表从当前 segments 去重生成：未知聚类显示编号，声纹命中显示姓名和部门/职位。
- 任何异步旧会话更新都不能修改新会议或导入详情。

## 5. 状态与错误处理

- 标题回声：发送 `code="context_echo_filtered"`，仅用于调试日志，不弹用户 toast。
- 声纹服务失败：文本照常落库，片段使用稳定匿名编号，页面不显示“转写失败”。
- embedding 太短或质量不足：保留当前匿名身份，后续更长片段可再次更新聚类质心。
- WebSocket 重连：会话从已有 segments 的 `speakerClusterId` 恢复编号计数，不能从“发言人1”重新开始并冲突。
- 导入转写继续使用已有离线 diarization + voiceprint 流程，不读取实时会话内存聚类。

## 6. 测试设计

### 6.1 后端

- 实时 context 不包含会议标题，但包含参与人和识别词。
- 最终文本仅等于会议标题时不落库、不增加 revision。
- 正常句子中包含标题片段时不被删除。
- 两个相近 embedding 复用“发言人1”，距离较远的 embedding 创建“发言人2”。
- 声纹库命中时覆盖匿名编号并保留部门/职位。
- embedding 服务失败仍落库文本，并分配稳定匿名编号。
- 重连后匿名编号不会重置或覆盖已有片段。

### 6.2 前端静态契约

- `meetingConfigSummary` 不再存在于 HTML，渲染函数不再调用。
- 搜索、筛选、保存状态均为中文。
- 五个 AI 工具按钮存在，标签为短中文文案。
- CSS 使用五列单行布局，按钮不换行，收起按钮不占网格列。
- `speaker_update` 只更新匹配的当前会议和 segment。

### 6.3 浏览器验证

1. 新建实时会议，安静 30 秒后逐字稿不能出现会议标题。
2. 一个人连续说两段，保持同一匿名编号。
3. 两个未登记人员交替说话，显示发言人1和发言人2。
4. 已登记声纹人员说话，显示声纹库姓名和部门/职位。
5. 图二调试条消失；图三为中文紧凑栏；五个 AI 工具在桌面与窄屏均不掉出或重叠。

## 7. 非目标

- 不更换 DashScope/Qwen ASR 模型。
- 不重新设计导入转写台账。
- 不伪造说话人身份或轮流分配编号。
- 不将声纹 embedding 发送给浏览器。
- 不借本次修复重构无关管理页面。
